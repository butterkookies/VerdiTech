VerdiTech Flow
Step 1 of 5

1. Plant Profile Setup

  • Select Crop: Tomato, Eggplant, Siling Labuyo
  • Set Date: Select planting date
  • Initial Stage: Seedling/Young Plant
Step 2 of 5

2. Daily Logging (Symptoms)

  • Frictionless Check: Rate Sun, Water, Soil (1-5)
  • Write Notes: Log specific issues or observations
  • Device Storage: Local saving in Drift database
Step 3 of 5

3. Time-Series CA Engine

  • Historical Sweep: Runs CA rules step-by-step from planting date to today using logged inputs
  • Future Projections: Simulates future steps based on historical averages
Step 4 of 5

4. Unified Timeline Display

  • Historical Cards: Solid colors showing logged stage/health drawings
  • Focus Item: Prominent "Today" card for fast updates
  • Forecast Cards: Dotted predicted stage timeline
Step 5 of 5

5. dynamic Advisories

  • Deficiencies analyzed dynamically
  • Generates specific recommendations based on crop, stage, and season

📄 lib/main.dart The application entry point. Initializes Flutter, sets up GoRouter with all 5 routes (/, /add-plant, /plant/:id, /plant/:id/visualization, /about), wraps the app in a Riverpod ProviderScope, and mounts VerdiTechApp with Material3 + Google Fonts (Inter).
Presentation Layer (5 Features)Contains all 5 feature screens organized by feature. Each feature lives in features/{name}/presentation/. Purely UI-focused — no business logic. Screens are ConsumerWidgets that watch Riverpod providers.
features/dashboardMain garden overview — watches plantsStreamProvider (Riverpod StreamProvider) and renders _PlantCard widgets with live health scores computed by the CA engine.
dashboard_screen.dartDisplays all plants via plantsStreamProvider. Each _PlantCard watches envScoreProvider and daysToHarvestProvider. FAB navigates to /add-plant. Tapping a card navigates to /plant/:id.
features/plant_formAdd-plant form screen. Uses PlantFormNotifier (StateNotifier) which manages form state and calls PlantRepository.addPlant() on submit.
add_plant_screen.dartForm with dropdowns for PlantType, GrowthStage, Season and sliders for sunlight/water/soil (1–5 scale). On save, calls plantFormProvider.notifier.save() which routes: PlantFormNotifier → PlantRepository → PlantDao → SQLite.
features/plant_detailAnalytics screen for a single plant. Watches forecastProvider (60-day CA timeline), recommendationsProvider, envScoreProvider, and daysToHarvestProvider — all computed live from plant state.
plant_details_screen.dartShows health card (envScore %), Predicted Growth Timeline (from forecastProvider mapping GrowthStage transitions), and Recommendations list. Action button navigates to /plant/:id/visualization.
features/ca_visualizationDedicated screen for visualizing raw 1D CA simulation output — shows the day-by-day GrowthStage grid that the prediction engine computes.
ca_visualization_screen.dartRenders the full CA forecast as a scrollable visual timeline grid. Watches forecastProvider(plant) and plots each DayPrediction's predictedStage and healthStatus. Accessible via the analytics icon on PlantDetailsScreen.
features/aboutStatic informational screen — no Riverpod providers or database access needed here.
about_screen.dartDescribes the app purpose, the 3 supported crop types (Tomato, Eggplant, Siling Labuyo), the Philippine seasonal model (Tag-init/Tag-ulan/Malamig), and the 1D Cellular Automata methodology.
State Management (Riverpod)The reactive glue layer. 3 provider files expose domain logic and data streams to the presentation layer. Screens never import the data layer directly — all data access flows through these providers.
providers3 provider files, each with a specific responsibility: database wiring, plant CRUD, and CA predictions.
database_provider.dartTwo singleton providers: databaseProvider (AppDatabase instance, auto-closed on ref.onDispose) and plantRepositoryProvider (PlantRepository backed by the database). ALL data access in the app flows through plantRepositoryProvider.
plant_providers.dartRead providers: plantsStreamProvider (real-time Stream<List<Plant>>), plantByIdProvider (FutureProvider.family), dailyLogsForPlantProvider (StreamProvider.family). Write: PlantFormNotifier (StateNotifier) manages add-plant form state and calls PlantRepository.addPlant() on save.
prediction_providers.dart6 pure computation providers (no DB calls): caEngineProvider, recommendationEngineProvider, environmentProfileProvider, envScoreProvider (0.0–1.0), forecastProvider (List<DayPrediction> for 60 days), daysToHarvestProvider, recommendationsProvider. All are Provider.family keyed by Plant.
Domain Layer (Core Logic)Pure Dart — zero Flutter or Drift imports. Contains the CA engine, recommendation engine, and 6 entity models. Fully unit-testable without a device or database.
domain/engine2 engines encapsulating all prediction and advisory logic. Stateless const classes — safe to share as Riverpod singletons.
ca_engine.dartCaEngine: computeEnvScore() weights sun/water/soil by PlantType and applies Philippine season modifiers. forecast() runs day-by-day GrowthStage transitions using minStageDays lookup tables and two thresholds (≥0.70 = progress, ≥0.40 = delayed, <0.40 = regression). estimateDaysToHarvest() runs a 180-day forecast to find harvestReady.
recommendation_engine.dartRecommendationEngine.generate() compares EnvironmentProfile scores against plant-specific optimal ranges for each GrowthStage, returning a prioritized List<Recommendation> covering watering, sunlight, and soil care.
domain/models6 pure Dart data classes — the shared vocabulary between all layers. Defined here, used everywhere.
plant.dartPlant entity: name, PlantType, DateTime plantingDate, GrowthStage currentStage, sunlightScore/waterScore/soilScore (double 1–5), Season, optional notes. Computed getter: daysPlanted = today − plantingDate.
daily_log.dartDailyLog entity for one day's care entry: plantId (FK), DateTime date, sunlightScore/waterScore/soilScore (double), optional notes. Stored in the daily_logs table with a unique (plantId, date) constraint for upserts.
enums.dartPlantType (tomato, eggplant, silingLabuyo), GrowthStage (seedling→youngPlant→flowering→fruiting→harvestReady with stageIndex), Season (tagInit/tagUlan/malamig with fromDate auto-detection), HealthStatus (critical→excellent via fromScore()), RecommendationPriority (high/medium/low).
environment_profile.dartLightweight value object: sunlight, water, soil (double), Season. Passed as input to CaEngine.computeEnvScore() and forecast(). Created by environmentProfileProvider from a Plant's current scores.
day_prediction.dartOne day's CA output: dayOffset (int, relative to planting), date, predictedStage (GrowthStage), healthStatus (HealthStatus), environmentalScore (double 0–1). The forecastProvider returns List<DayPrediction> covering 60 days.
recommendation.dartSingle advisory: category (string e.g. 'Watering'), message (human-readable tip), priority (RecommendationPriority). The recommendationsProvider returns List<Recommendation> displayed in PlantDetailsScreen.
Data Layer (Persistence)All SQLite/Drift persistence lives here. Screens never import this layer directly — access flows through Riverpod providers → PlantRepository → DAOs → Mappers → Tables.
data/repositoriesPlantRepository is the single public API of the data layer. Providers call repository methods; the repository delegates to the appropriate DAO.
plant_repository.dartPlantRepository(AppDatabase): watchAllPlants(), addPlant() → inserts and returns Plant with new id, updatePlant(), deletePlant() (cascade-deletes logs first via dailyLogDao.deleteLog()), watchLogsForPlant(), upsertLog() (idempotent daily care save), deleteLog().
data/databaseCore Drift database wiring: @DriftDatabase annotation registers PlantTable + DailyLogTable and PlantDao + DailyLogDao. Opens verditech.sqlite in the app documents directory.
app_database.dartAppDatabase extends _$AppDatabase. Opens a LazyDatabase at verditech.sqlite via path_provider. Schema version 1, auto-creates all tables on first launch via onCreate: m.createAll().
data/database/tables2 Drift Table schema definitions — the source of truth for all SQLite column types and constraints.
plant_table.dartPlantTable: id (PK auto-increment), name (text), type (PlantType enum as string), plantingDate (ISO-8601 text), currentStage (GrowthStage enum as string), sunlightScore/waterScore/soilScore (real), season (Season enum as string), notes (nullable text), createdAt (ISO-8601 text).
daily_log_table.dartDailyLogTable: id (PK auto-increment), plantId (FK → plants.id), date (ISO-8601 text), sunlightScore/waterScore/soilScore (real), notes (nullable text). UNIQUE(plantId, date) enables upsertLog() to replace an existing entry for the same day.
data/database/daos2 @DriftAccessor classes — the only layer that writes Drift query expressions. Each DAO handles one table's full typed query set.
plant_dao.dartPlantDao (@DriftAccessor on PlantTable): watchAllPlants() Stream ordered desc by createdAt, getAllPlants(), getPlantById(), insertPlant() → returns new int id, updatePlant() (replace), deletePlant(). All rows pass through PlantMapper.fromRow() before being returned as domain Plants.
daily_log_dao.dartDailyLogDao (@DriftAccessor on DailyLogTable): watchLogsForPlant() Stream ordered by date ASC, getLogsForPlant(), insertLog(), updateLog(), deleteLog(), upsertLog() (insertOnConflictUpdate — safe to call multiple times for the same date). Rows pass through DailyLogMapper.
data/mappers2 stateless mapper classes — the anti-corruption layer between Drift-generated row data classes (which import Drift) and clean domain model classes (which must not import Drift).
plant_mapper.dartPlantMapper.fromRow(PlantTableData) → Plant domain model (parses ISO-8601 strings → DateTime, enum strings → enum values). PlantMapper.toCompanion(Plant) → PlantTableCompanion (serializes enums → strings, DateTime → ISO-8601) for Drift insert/update calls.
daily_log_mapper.dartDailyLogMapper.fromRow(DailyLogTableData) → DailyLog domain model. DailyLogMapper.toCompanion(DailyLog) → DailyLogTableCompanion for Drift upsert operations. Handles nullable notes and ISO-8601 date serialization.
